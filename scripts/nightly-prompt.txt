## Step 1 — Find or create work

Run `openspec list --json` to check for active (non-archived) changes.

**If an active change exists with pending tasks:** skip to Step 2 using that change.

**If all active changes report state "all_done" but have NO corresponding git branch or PR:** treat them as stale — their tasks.md was pre-checked without real implementation. Reset by running `openspec status --change "<name>" --json` and checking whether the code files listed in the tasks actually exist and the test suite passes. If tests fail or files are missing, uncheck the relevant tasks (`- [x]` → `- [ ]`) and proceed to Step 2.

## Step 2 — Implement

Run `openspec instructions apply --change "<name>" --json` to get pending tasks.

**Before picking a task, cross-check open PRs.** Yesterday's nightly may have opened a PR for a task that is still unchecked in `tasks.md` because the PR is awaiting human review. Implementing the same task again produces a duplicate PR with conflicting `tasks.md` edits.

Run `gh pr list --state open --json number,title,body,headRefName --limit 50`. Apply two checks against the result, in order:

1. **Whole-change match.** Compare `<change-name>` against every open PR's `headRefName` exactly. By convention the branch for a single-task change matches the change directory (`add-reset-syntax`, `fix-mps-encoding-non-factorizing`, …), so an exact match means the entire change is already in flight as one PR. This catches changes that do NOT use `§N.M`-style anchors and would otherwise slip past the per-task scan below. If any open PR has `headRefName == <change-name>`, stop and report `"Change <change-name> already covered by open PR #<n>."` — do not implement anything; do not open a PR.

2. **Per-task match.** For each pending task, scan every open PR's title, body, and `headRefName` for the task's section identifier (e.g. `§4.3`, `4.3 `, `tech-debt-backlog-4-3`). If any open PR already covers the task, treat that task as taken and pick the next one. This is the granular case for multi-task changes like `tech-debt-backlog`, where individual PRs branch off as `tech-debt-backlog-7-18`, `tech-debt-backlog-7-16-7-17`, etc.

If every pending task is already covered by an open PR, stop and report `"No uncovered work — N PRs await human review."` Do not implement anything; do not open a PR.

Work through each remaining pending task. Keep changes minimal and focused.

**CRITICAL: Only mark a task complete (`- [ ]` → `- [x]`) AFTER:**
1. The code change has been written to disk (Read the file to confirm)
2. `pytest` passes (run the full suite or the relevant test file)

Never pre-check tasks. A task is done when the tests prove it, not before.

## Step 4 — Open PR

Run `gh pr create` with a title and body summarising what was implemented and what tests were added.

## Step 5 — Wait for CI (BOUNDED — do not block forever)

Poll CI with a hard ceiling. Do NOT use `gh pr checks --watch` — it has no timeout and will block forever if a check hangs. Instead, run this bounded loop (≤ 10 minutes; ~20 checks at 30s each):

```bash
PR=<number>
for i in {1..20}; do
  status=$(gh pr checks "$PR" --json bucket --jq '.[].bucket' 2>/dev/null || echo "error")
  if [[ -z "$status" ]] || ! echo "$status" | grep -q "pending"; then
    break
  fi
  sleep 30
done
gh pr checks "$PR" || true
```

If after the loop any check still reports pending, log the state and proceed to Step 6 — don't block further. If checks failed, diagnose and push fixes (one round of fix-and-repush is fine; don't chase indefinitely).

## Step 5.5 — Bail conditions

Stop the run early if any of these are true:
- The fix-and-repush loop has executed more than 2 cycles (CI is in a bad state; flag for human review).
- A single tool call has been running more than 5 minutes without completing.
- Disk write would exceed 100 MB (something is wrong with the diff).

Print the bail reason in the summary and stop.

## Step 6 — Report and stop

Print a final summary:
- Change name and what was implemented
- PR URL
- Tasks completed
- CI status

Do NOT merge the PR. Leave it open for human review.
